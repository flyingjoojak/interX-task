# ADR — Architecture Decision Records
# InterX 채용 솔루션

각 결정은 **결정 → 이유 → 트레이드오프** 순으로 기술.
이 문서를 통해 "왜 이 기술인가"를 추론 가능하도록 맥락 보존.

## ADR-001. 데이터베이스: SQLite 선택

**결정:** PostgreSQL 대신 SQLite 사용.

**이유:**
- 로컬 단일 머신 배포, 동시 쓰기 부하 없음
- 설정 없이 파일 하나로 즉시 실행 → MVP 안정성 우선
- SQLAlchemy ORM 사용으로 PostgreSQL 전환 시 연결 문자열만 변경

**트레이드오프:**
- pgvector(시맨틱 검색) 사용 불가 → MVP 범위 밖이므로 수용
- 향후 다중 사용자/배포 시 PostgreSQL 마이그레이션 필요

## ADR-002. OCR 파이프라인: 3단계 폴백 구조

**결정:** 이력서는 PyMuPDF → PaddleOCR → Claude Vision API 순 폴백.

**이유:**
- 텍스트 기반 PDF는 PyMuPDF로 완벽 추출 (빠르고 무료)
- 스캔본/이미지 PDF는 PaddleOCR (한국어 특화, 로컬, 무료)
- 품질이 여전히 낮으면 Claude Vision (최고 품질, 유료)
- 비용 최소화 + 품질 보장의 균형

**트레이드오프:**
- 3단계 처리로 로직 복잡도 증가 → quality_score 임계값(0.7)으로 단순화
- Claude Vision API 비용 발생 → 폴백이므로 빈도 낮음

## ADR-003. 포트폴리오 처리: 전량 이미지 → Claude Vision

**결정:** 포트폴리오는 형식에 상관없이 이미지로 변환 후 Claude Vision API 처리.

**이유:**
- 포트폴리오는 디자인/레이아웃 정보가 핵심 (텍스트만 추출 시 정보 손실)
- Claude Vision은 시각적 레이아웃, 차트, 이미지 내 텍스트 모두 이해
- 파이프라인 단순화: 모든 포트폴리오를 동일하게 처리

**트레이드오프:**
- 이력서보다 Claude Vision API 사용량 증가 → 포트폴리오는 1개이므로 수용
- PPT/PPTX 슬라이드 렌더링은 LibreOffice 필요 → 의존성 회피를 위해 PPT/PPTX 미지원 결정

## ADR-004. PPT/PPTX 포트폴리오 미지원

**결정:** 포트폴리오에서 PPT/PPTX 형식 미지원. PDF/JPG/PNG만 허용.

**이유:**
- 슬라이드 전체 레이아웃을 이미지로 변환하려면 LibreOffice 설치 필요
- 외부 소프트웨어 의존성은 MVP 안정성을 해침
- python-pptx는 텍스트만 추출하여 포트폴리오 의도 파악 불가

**트레이드오프:**
- 일부 사용자는 PPT 포트폴리오 보유 → 업로드 존에 "PDF 변환 권장" 안내로 대응
- 이력서의 PPT/PPTX는 텍스트 중심이므로 python-pptx로 충분 → 이력서만 허용 유지

## ADR-005. AI 모델: Claude claude-sonnet-4-6 API 선택

**결정:** 로컬 모델(Ollama/Llama) 대신 Claude claude-sonnet-4-6 API 사용.

**이유:**
- 한국어 이력서 분석 정확도: Claude >> 로컬 모델
- 12가지 가치 매핑, 모순 탐지, 압박질문 생성 모두 복잡한 추론 필요
- MVP 목표는 정확도 검증이므로 최고 품질 모델 우선

**트레이드오프:**
- 이력서 1건당 약 $0.05~0.15 API 비용 발생
- 데이터 외부 전송 발생 → PII 익명화로 완화
- Anthropic은 API 호출 데이터를 학습에 사용하지 않음 (공식 정책)

## ADR-006. 에이전트 프레임워크: LangGraph 선택

**결정:** 단순 LLM 호출 대신 LangGraph로 에이전트 워크플로우 구성.

**이유:**
- 사전 분석이 8단계 순차 처리 → 단계별 상태 관리 필요
- 각 단계에서 DB 업데이트(진행률) + 조건 분기 + 오류 처리 필요
- LangGraph의 State 기반 그래프가 이 구조에 최적

**트레이드오프:**
- LangChain/LangGraph 학습 비용 → 공식 문서화 잘 되어 있음
- 오버엔지니어링 위험 → 8단계 파이프라인은 단순 함수 체인 대비 LangGraph가 적합

## ADR-007. AI 생성 답변 탐지 → Drill-down 질문으로 대체

**결정:** AI 탐지기 사용 안 함. 대신 검증이 어려운 답변에 심층 꼬리질문 생성.

**이유:**
- 현존 AI 탐지기 오탐률 15~30% → 합불 판단에 사용 시 법적/신뢰 리스크
- "AI가 썼는지"보다 "실제 경험이 있는지"가 채용 목적에 더 부합
- 실제 경험자는 구체적 drill-down 질문에 답변 가능, AI 생성 답변자는 불가

**트레이드오프:**
- AI 탐지 기능 없음 → 면접관에게 명시적으로 안내 필요
- 오히려 더 강력한 검증 방식 (직접 증거 확인)

## ADR-008. 인증: JWT 계정 방식

**결정:** 단순 비밀번호 보호 대신 JWT 면접관 계정 방식.

**이유:**
- 복수 면접관이 동일 서버 접속 시 개인 식별 필요
- 향후 면접관별 활동 로그, 권한 관리 확장 고려
- 로컬이더라도 브라우저 접근이므로 최소한의 인증 필요

**트레이드오프:**
- 구현 복잡도 증가 → python-jose로 단순화
- 계정 관리 화면 미구현 → 초기 시드 스크립트 + 개발자 직접 추가로 MVP 수용

## ADR-009. 분석 진행 상태: 폴링 방식

**결정:** Server-Sent Events(SSE) 대신 2초 폴링 방식.

**이유:**
- SSE는 서버 연결 유지 + 연결 끊김 처리 로직 필요 → 복잡도 증가
- 2초 폴링은 구현 단순, 디버깅 쉬움, 실패 시 자동 재시도
- 분석 소요 시간(~60초)에서 2초 지연은 UX 허용 범위

**트레이드오프:**
- 폴링 주기마다 HTTP 요청 발생 (~30회/분석) → 로컬 환경에서 무의미한 부하
- UX 반응성이 SSE보다 약간 낮음 → 진행률 바로 충분히 커버

## ADR-010. PDF 출력: Playwright 서버 사이드 렌더링

**결정:** react-pdf, jsPDF 대신 Playwright로 서버에서 브라우저 렌더링 후 PDF 생성.

**이유:**
- 대시보드에서 보이는 화면을 그대로 PDF로 출력 → 별도 레이아웃 작업 불필요
- 한국어 폰트, Recharts 차트 모두 완벽 렌더링
- 사전 분석 리포트 + 면접 세션 통합 PDF를 하나의 엔드포인트로 처리

**트레이드오프:**
- Playwright Chromium 설치 필요 (~300MB) → 일회성 설치, 이후 안정적
- 클라이언트 사이드 PDF보다 서버 자원 소모 → 로컬 환경에서 무시 가능

## ADR-011. 면접 Q&A 구조: 선형 타임라인

**결정:** 트리 구조 대신 시간순 선형 구조로 Q&A 저장.

**이유:**
- 면접관은 꼬리질문을 선택하거나 다른 질문으로 이동하지만 대화는 시간순 진행
- 선형 구조가 화면 표시(채팅형), 데이터 조회, PDF 출력 모두 단순
- parent_qa_id는 컨텍스트 참조용으로만 사용 (트리 탐색 불필요)

**트레이드오프:**
- 질문 분기 구조가 DB에 명시적으로 표현되지 않음 → order_index + parent_qa_id로 충분

## ADR-012. 상태 관리: 수동/자동 분리

**결정:** 시스템 자동 상태(미분석/분석중/분석완료)와 면접관 수동 상태(합격/탈락) 명확 분리.

**이유:**
- 자동 상태는 파이프라인 제어용 → 면접관이 건드리면 안 됨
- 수동 상태는 순수 메모/표기용 → 어느 화면에서든 자유롭게 변경 가능
- 상태 변경이 다른 기능에 영향을 주지 않음 (단순 레이블)

**트레이드오프:**
- 상태 전이에 비즈니스 로직 없음 → 잘못된 상태 설정 가능 (서류탈락인데 면접합격 등)
- 의도적 결정: 면접관 재량 최대화, 시스템은 보조 도구
