## 슬라이드 5 — 지원 동기

**헤드**: 왜 **인터엑스**여야 하는가

**3포인트 + 인용문 스타일**
- **"AI 에이전트로 채용을 재정의"** — 단순 ATS 자동화가 아니라, 사람의 판단을 **정량적 근거와 이중 검증**으로 보조하는 방향. 제가 꾸준히 추구해온 "효율로 증명되는 AI"와 정확히 같은 지향.
- **정형화되지 않은 문제** — 이력서 환각, 면접 대화의 모순 탐지, 설명 가능한 AI 결과. 제가 RAG에서 토큰·정확도 트레이드오프를 직접 설계했던 경험과 맞닿음.
- **AX(AI Experience) 관점의 프로덕트** — 모델 성능 그 자체보다 **면접관·지원자에게 실제 가치**로 연결되는 지점을 설계하는 문화. 이 관점은 제가 커뮤니티 봇에서 실제 사용자 DAU를 숫자로 검증하며 체득한 부분.

**스피커 노트**: AI 회사는 많지만, "AI를 써서 무엇을 바꾸는가"에 답이 있는 회사는 드묾. 인터엑스는 그 답이 "채용이라는 고위험·정성 업무의 구조화"라는 점이 제 관심사와 가장 정확히 겹침.

## 슬라이드 8 — 주제 1. 데이터 파이프라인

**헤드**: 비정형 PDF → 구조화 JSON → 12가치 매핑

**왼쪽 컬럼 — 파이프라인 단계**:
1. **OCR** — Claude Vision 기반. 읽는 순서 강제(위→아래, 좌→우, 2컬럼 블록 유지) 프롬프트.
2. **익명화** — 이름·이메일·전화번호 토큰화(PII-safe), 분석 후 복원.
3. **구조화 추출** — career/education/skills/achievements/certifications/projects 6필드 JSON. 날짜 포맷·재직중 처리 규칙 내재화.
4. **12가치 매핑** — 각 가치에 0–100 점수 + `evidence`(1–2문장 요약) + `examples`(원문 인용 배열).

**오른쪽 컬럼 — 프롬프트 엔지니어링의 디테일**:
- "추상적 자기서술('열정적입니다' 등)은 증거로 인정하지 않음" — 근거 강제.
- 80점 기준을 프롬프트 상수로 고정("반복적·정량적 증거가 있을 때만").
- 결과는 `Analysis.values_scores` 컬럼 + 프론트 `ValueRadarChart`·`EvidencePanel` 로 원문 인용과 같이 시각화.

**배치**: 오른쪽 아래 `report.png` (12가치 레이더 차트 실제 화면)을 썸네일로.

**스피커 노트**: 단순히 "Claude에게 시켜서 매핑"이 아니라, 프롬프트에 **부정 예시와 거절 기준**을 명시해 모델이 추상 자기서술에 높은 점수를 주는 실패를 체계적으로 차단.

## 슬라이드 9 — 주제 2. 이중 검증 면접 에이전트

**헤드**: LangGraph 두 개로 설계한 모순 탐지 + 꼬리질문 체인

**레이아웃**: 좌측 다이어그램, 우측 실제 화면 3장(`모순.png`, `사전압박질문.png`, `면접.png`).

**체인 ① — 분석 그래프 (사전)**:
```
parse_documents → anonymize_pii → extract_structured_data
    → score_12_values → self_verify_evidence
    → calculate_doc_reliability → detect_contradictions
    → generate_preemptive_questions → compile_and_restore
```
- 4축 모순 탐지: 날짜/기간·직책/역할·성과 수치·기술 스택. severity high/medium/low.
- 사전 압박 질문: 모순 1건당 1문항 이상 + 40점 미만 가치 드릴다운. 개방형 질문 금지("~에 대해 말씀해주세요" 차단).

**체인 ② — 면접 그래프 (실시간)**:
```
prepare_context → analyze_answer → generate_followups → rank_and_filter
```
- `question_source` (pregenerated/custom/followup) 별 분기 — 꼬리질문의 꼬리질문도 생성 생략하지 않도록 프롬프트에 출처 강제.
- asyncio.Queue 단일 워커로 LLM 호출 직렬화 → 다중 답변 제출에도 순서 꼬임·중복 생성 방지.

**스피커 노트**: "추론 로직"을 프롬프트 한 번에 맡기지 않음. 노드 단위로 책임 분리 → 한 노드가 실패해도 `current_step` 에 남아 재개 가능. 이 구조 덕에 프롬프트 수정·회귀 테스트도 노드별로 떼어낼 수 있음.

## 슬라이드 11 — 엔지니어링 고민 ① Hallucination 방지

**헤드**: "근거가 원문에 존재하는가"를 기계가 스스로 확인하게 만들었습니다

**문제 정의**: LLM이 `examples` 필드에 **원문에 없는 문장**을 만들어 넣는 환각. 면접관이 리포트를 신뢰할 수 없게 되는 치명적 실패 모드.

**방어선 4층**:
1. **프롬프트 제약** — "추상적 자기서술은 증거 불인정", "원문 문장 그대로 인용".
2. **근거 필드 강제** — 12가치 `evidence`/`examples`, 사전질문 `basis`, 꼬리질문 `reasoning` 전부 스키마에 필수.
3. **자가 점증 검증 루프** *(핵심)*
   - 각 `examples` 에 대해 이력서 원문과 `substring` + `token-overlap ≥ 0.6` 두 축으로 검증.
   - 점수 40↑ & 검증 비율 < 50% → 해당 가치만 `VALUE_REGENERATION_PROMPT` 로 1회 재호출해 인용 교체·점수 보정.
   - 결과는 `verification = { verified_examples, unverified_examples, verified_ratio, regenerated? }` 로 DB 영속화.
4. **사람 검증 UI** — `RawDataViewer` 로 OCR 원문 ↔ 구조화 JSON 을 나란히 비교.

**다이어그램**: `score_12_values → self_verify_evidence(verify | regenerate) → calculate_doc_reliability`

**스피커 노트**: "프롬프트로 당부했다"와 "기계가 체크한다"는 다른 층위. 이번 과제의 가장 큰 엔지니어링 결정.

## 슬라이드 12 — 엔지니어링 고민 ② 토큰·비용 실측

**헤드**: 대략치 메모가 아니라 **DB에 남는 수치**로 비용을 본다

**설계**:
- 신규 테이블 `token_usage(candidate_id, phase, step, model, input_tokens, output_tokens, cache_read, cache_write, cost_usd, created_at)`.
- `contextvars.ContextVar` 기반 `usage_scope(candidate_id, phase, step)` — LangGraph 노드·OCR·Vision·면접 에이전트 호출이 **자동으로 phase/step 태깅**되어 기록.
- `GET /api/candidates/{id}/cost` — 총 호출 수·입/출력 토큰·USD 추정 비용 + phase/step 분해.
- 리포트 페이지 "Claude API 사용량 · 비용" 섹션에서 면접관이 즉시 확인.

**차트 (실측 기반, 측정 후 삽입)**:
- **바 차트 A** — phase별 USD 비용 (analysis / interview).
- **바 차트 B** — step별 입·출력 토큰 (OCR · 추출 · 가치매핑 · 모순탐지 · 질문생성).
- **수치 박스** — 1회 분석 총 USD / 총 토큰 / 호출 수.

> 실측치는 `/api/candidates/{id}/cost` 응답으로 대체. (슬라이드 작성 시 placeholder: `$0.XXXX · XXXX tokens · N calls`)

**스피커 노트**: "비용 최적화에 신경 썼습니다"를 말로만 하지 않고, **다음 PR에서 얼마를 깎았는지 볼 수 있는 계측 인프라**를 먼저 넣었다는 점이 포인트. 이것이 있어야 최적화 실험이 가능.
