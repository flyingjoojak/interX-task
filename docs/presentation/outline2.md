## 슬라이드 4 — 지원 동기

**헤드**: 왜 **인터엑스**여야 하는가

**3포인트 + 인용문 스타일**

- **"AI 에이전트로 채용을 재정의"**
  단순 ATS 자동화가 아니라, 사람의 판단을 **정량적 근거와 이중 검증**으로 보조하는 방향. 제가 꾸준히 추구해온 "효율로 증명되는 AI"와 정확히 같은 지향.

- **정형화되지 않은 어려운 문제**
  이력서 환각, 면접 대화의 모순 탐지, 설명 가능한 AI 결과 — 전부 RAG 시스템에서 토큰·정확도 트레이드오프를 직접 설계해 본 제 경험이 그대로 쓰일 수 있는 영역.

- **AX(AI Experience) 관점의 프로덕트**
  모델 성능 그 자체보다 **면접관·지원자에게 실제 가치**로 연결되는 지점을 설계하는 문화. 디스코드 커뮤니티 봇에서 실사용자 DAU 300·첫주 매출 49만원으로 검증해 본 "현장 적합도" 감각과 직결.

**스피커 노트**:
AI 회사는 많지만 "AI를 써서 무엇을 바꾸는가"에 답이 있는 회사는 드뭅니다. 인터엑스는 그 답이 "채용이라는 고위험·정성 업무의 구조화"라는 점이 제 관심사와 가장 정확히 겹칩니다.

## 슬라이드 5 — 12가치 매칭 + 입사 후 포부

**헤드**: 12가치 중 4개에 먼저 기여하겠습니다

**12가치 전체 리스트(작게 뒤에 배치)** → 그 위에 **집중할 4가지**를 강조:

| 가치 | 제 이력의 근거 |
|---|---|
| **끈기** | 약제 식별 프로젝트에서 **750회 이상의 반복 실험과 정밀 튜닝**을 거쳐 인식 정확도 **99.1%** 기록. |
| **지속적개선** | RFP RAG에 **FlashRank 리랭킹** 도입으로 노이즈 제거 + 연관성 높은 컨텍스트 선별. **MobileNetv2 기반 RapidOCR**을 ONNX로 최적화해 부하 최소화 및 추론 구현. **양자화·어텐션 최적화**로 하드웨어 추론 성능 확보. |
| **문제해결** | 조제실 내 **오투약 리스크**를 줄이기 위한 약제 식별 시스템 구축. 외국어 게임/영상 시청 시 발생하는 **정보 격차**를 실시간 번역 오버레이로 해소. **클릭 간격 표준편차** 기반 행동 패턴 알고리즘으로 매크로 방지 로직 적용. |
| **비판적사고** | 단순 LLM 호출에서 벗어나 **Hybrid Search + FlashRank 리랭킹**으로 토큰 낭비 절감과 신뢰도 확보. 운영비 절감을 위해 유료 API 대신 **로컬 허깅페이스(SDXL) 모델 및 서빙 환경** 구축. **YOLOv11과 Faster R-CNN을 비교**하고 Optuna로 하이퍼파라미터 최적화 수행. |

"이 네 가치를 일상의 코드 레이어에 — 회귀 테스트·비용 계측·자가 검증 루프로 — 박아 넣어, 분기마다 더 싸지고 더 정확해지는 시스템을 만들겠습니다."

**스피커 노트**:
12가치를 다 잘한다는 주장은 거짓말입니다. 가장 빠르게 기여할 수 있는 4개에 집중하겠습니다. 이 네 가치는 이번 과제에서도 그대로 드러납니다 — 끈기는 검증 루프 반복 설계로, 지속적개선은 토큰 계측 인프라로, 문제해결은 모순 탐지 4축 설계로, 비판적사고는 모델 출력을 그대로 믿지 않는 자가 검증 레이어로.

## 슬라이드 6 — 과제 오버뷰

**헤드**: InterX MVP — FastAPI · LangGraph · Next.js 풀스택

**배치**: 왼쪽 `home.png` 스크린샷, 오른쪽 플로우 다이어그램.

**플로우 다이어그램(오른쪽)**:
```
① 이력서/포트폴리오 업로드 (PDF·JPG·PNG)
        ↓  Claude Vision OCR
② 구조화 추출  →  ③ 12가치 매핑
        ↓  자가 검증 루프 (substring + token-overlap)
④ 문서 신뢰도  →  ⑤ 모순 탐지  →  ⑥ 사전 압박 질문
        ↓
⑦ 실시간 면접 (LangGraph 꼬리질문 체인)
        ↓
⑧ PDF 리포트 · 면접관 메모 · 비용 리포트
```

**정량 스펙**:
- 분석 파이프라인 **8단계** (LangGraph StateGraph)
- 면접 꼬리질문: **asyncio.Queue 단일 워커**로 LLM 호출 직렬화
- 저장소: SQLite + reportlab PDF (한글 폰트 자동 등록)
- 모든 LLM 호출은 `usage_scope` 컨텍스트로 **phase/step 자동 태깅**

**스피커 노트**:
세 미션(데이터 / 로직 / 인터페이스)을 모두 한 흐름으로 묶어 실제 동작하는 풀스택으로 만들었습니다. 화면 왼쪽이 실제 메인 화면이고, 오른쪽이 데이터의 흐름입니다.

## 슬라이드 7 — 미션 1. 데이터 파이프라인 [수행]

**헤드**: 비정형 PDF/이미지 → 구조화 JSON → 12가치 매핑 — 4단계 파이프라인

**왼쪽 컬럼 — 파이프라인 단계 (`backend/agents/analysis_graph.py`)**:
1. **OCR** — Claude Vision 기반(`services/ocr_service.py`). 페이지 단위로 이미지화 후 "위→아래, 좌→우, 2-컬럼 블록 유지" 읽기 순서를 프롬프트로 강제.
2. **익명화 (PII-safe)** — 이름·이메일·전화번호를 토큰으로 치환해 LLM에 전달, 분석 후 복원. 프롬프트가 개인정보로 오염되는 위험 차단.
3. **구조화 추출** — `STRUCTURED_EXTRACTION_PROMPT`로 6필드 JSON 변환: `career / education / skills / achievements / certifications / projects`. 날짜 포맷·"재직중" 처리 규칙을 프롬프트 내 상수로 고정.
4. **12가치 매핑** — `VALUE_SCORING_PROMPT`로 가치별 `score(0–100) / evidence(1–2문장) / examples(원문 인용 배열)` 생성.

**오른쪽 컬럼 — 프롬프트 엔지니어링 디테일**:
- **부정 예시 명시**: "추상적 자기서술('열정적입니다', '책임감이 강합니다')은 증거로 인정하지 않음".
- **80점 컷 고정**: "충분한 구체적 증거(수치·프로젝트·행동 결과)가 반복적으로 관찰되는 경우"에만 80↑.
- **거절 기본값**: 증거가 없는 가치는 0–30 부여하고 `examples`는 빈 배열.
- **원문 인용 강제**: `examples`는 이력서의 구체 문장을 **원문 그대로** 1–3개 인용.

**스피커 노트**:
"Claude에게 시켜서 매핑한 코드"가 아니라, **부정 예시와 거절 기준을 프롬프트에 박아 넣어** 모델이 자기서술에 높은 점수를 주는 실패 모드를 체계적으로 차단했다는 점이 포인트.

## 슬라이드 8 — 미션 1. 데이터 파이프라인 [결과]

**헤드**: 구조화 데이터셋 + 12가치 점수 + 원문 증거가 한 객체로

**결과물 (구조화 데이터)**:
- DB 스키마: `Analysis.structured_data` (6필드 JSON), `Analysis.values_scores` (12가치별 점수+근거).
- API: `GET /api/candidates/{id}/analysis` — 분석 결과 통째로 JSON 반환.
- API: `GET /api/candidates/{id}/debug/raw` — OCR 원문 ↔ 구조화 데이터 묶어 반환.

**결과물 (시각화)**:
- `ValueRadarChart` — 12가치 점수를 레이더 차트로 한눈에.
- `ValueList` — 가치 클릭 시 `EvidencePanel` 슬라이드아웃에 원문 증거 노출.
- `RawDataViewer` — OCR 원문과 구조화 JSON을 탭 UI로 나란히 비교.

**시각**: 우측에 `report.png` 썸네일(레이더 차트 실제 화면).

**예시 출력 (요약)**:
```json
{
  "values_scores": {
    "지속적개선": {
      "score": 88,
      "evidence": "ONNX 가속·양자화·리랭킹 도입을 통한 반복적 효율 개선이 명시.",
      "examples": ["ONNX 가속으로 추론 지연 시간 70% 단축",
                   "Reranking 도입으로 비용 30% 절감"]
    }
  }
}
```

**스피커 노트**:
면접관이 "이 점수는 어디서 나온 건가요?"를 물을 때, 화면에서 가치 한 번 클릭하면 **이력서 원문 인용**이 바로 따라 뜹니다. 점수 자체가 아니라 점수의 **추적 가능성**이 결과물의 핵심.

## 슬라이드 9 — 미션 2. 모순 탐지 + 꼬리질문 [수행]

**헤드**: LangGraph 두 개로 설계한 사전 분석 + 실시간 면접 체인

**체인 ① — 분석 그래프 (사전, `analysis_graph.py`)**:
```
parse_documents → anonymize_pii → extract_structured_data
    → score_12_values → self_verify_evidence
    → calculate_doc_reliability → detect_contradictions
    → generate_preemptive_questions → compile_and_restore
```
- **모순 탐지 4축** (`CONTRADICTION_DETECTION_PROMPT`): 날짜/기간 · 직책/역할 · 성과 수치 · 기술 스택. severity `high/medium/low`.
- **사전 압박 질문** (`PREEMPTIVE_QUESTIONS_PROMPT`): 모순 1건당 1문항 이상 + **40점 미만 가치** 드릴다운. **개방형 질문 금지** ("~에 대해 말씀해주세요" 차단), 반드시 수치·시점·결정자를 묻도록 제약.

**체인 ② — 면접 그래프 (실시간, `interview_graph.py`)**:
```
prepare_context → analyze_answer → generate_followups → rank_and_filter
```
- `prepare_context`: 구조화 데이터 + 저점 가치 + 최근 답변 3개 로딩.
- `analyze_answer`: 답변의 모호성/불일치/과장 여부 선분석.
- `generate_followups`: 위 분석을 근거로 꼬리질문 후보 생성.
- `rank_and_filter`: priority 정렬, 상위 5개 채택.
- **`question_source` 분기** (`pregenerated/custom/followup`): 꼬리질문의 꼬리질문도 생략하지 않도록 프롬프트에 출처 강제.
- **`asyncio.Queue` 단일 워커** (`services/followup_worker.py`): 동시 답변 제출에도 LLM 호출이 직렬화되어 순서 꼬임·중복 차단.

**스피커 노트**:
"추론 로직"을 프롬프트 한 번에 맡기지 않았습니다. 노드 단위로 책임을 분리해 한 노드가 실패해도 `current_step`에 남아 **재개 가능**합니다. 이 구조 덕에 프롬프트 수정·회귀 테스트도 노드별로 떼어낼 수 있습니다.

## 슬라이드 13 — 엔지니어링 고민 ① 할루시네이션 자가검증 루프

**헤드**: "근거가 원문에 존재하는가"를 기계가 스스로 확인하게 만들었습니다

**문제 정의**:
LLM이 `examples` 필드에 **원문에 없는 문장**을 만들어 넣는 환각 — 면접관이 리포트를 신뢰할 수 없게 되는 치명적 실패 모드.

**4층 방어선**:
1. **프롬프트 제약** — "추상적 자기서술은 증거 불인정", "원문 문장 그대로 인용".
2. **근거 필드 강제** — 12가치 `evidence`/`examples`, 사전질문 `basis`, 꼬리질문 `reasoning` 전부 스키마에 필수.
3. **자가 점증 검증 루프** *(핵심)*
   - 각 `examples`에 대해 이력서 원문과 `substring` + `token-overlap ≥ 0.6` 두 축으로 검증.
   - **점수 40↑ & 검증 비율 < 50% → 해당 가치만 `VALUE_REGENERATION_PROMPT`로 1회 재호출**해 인용 교체·점수 보정.
   - 결과는 `verification = { verified_examples, unverified_examples, verified_ratio, regenerated? }`로 DB 영속화.
4. **사람 검증 UI** — `RawDataViewer`로 OCR 원문 ↔ 구조화 JSON을 나란히 비교.

**다이어그램**: `score_12_values → self_verify_evidence(verify | regenerate) → calculate_doc_reliability`

**스피커 노트**:
"프롬프트로 당부했다"와 "기계가 체크한다"는 다른 층위입니다. 이번 과제에서 가장 큰 엔지니어링 결정이고, **검증 결과 자체를 DB에 남겨** 다음 프롬프트 개선의 입력으로 쓸 수 있게 만들었습니다.

## 슬라이드 14 — 엔지니어링 고민 ② 토큰·비용 실측 + Claude Code 활용

**헤드**: 대략치 메모가 아니라 **DB에 남는 수치**로 비용을 본다 — 그리고 그 수치를 만든 도구도 정직하게.

**비용 계측 설계**:
- 신규 테이블 `token_usage(candidate_id, phase, step, model, input_tokens, output_tokens, cache_read, cache_write, cost_usd, created_at)`.
- `contextvars.ContextVar` 기반 `usage_scope(candidate_id, phase, step)` — LangGraph 노드·OCR·Vision·면접 에이전트 호출이 **자동으로 phase/step 태깅**되어 기록.
- `GET /api/candidates/{id}/cost` — 총 호출 수·입/출력 토큰·USD 추정 비용 + phase/step 분해.
- 리포트 페이지 **"Claude API 사용량 · 비용"** 섹션에서 면접관이 즉시 확인.

**측정값 (참고)**:
- 분석 1회 추정: **$0.15–0.30**
- 꼬리질문 1회 추정: **$0.03–0.05**

**최신 AI 도구 활용 — Claude Code**:
- **Claude Code (claude-opus-4-7)** 를 페어 프로그래밍 파트너로 사용. 단순 코드 생성이 아니라 **설계 단계의 비판적 검토 도구**로 활용.
- 적용 범위:
  - LangGraph 노드 분리 설계 검토
  - 자가검증 루프의 substring/token-overlap 임계값 결정 토론
  - reportlab 한글 폰트 등록 트러블슈팅
  - 본 발표 자료(스크립트·차트 명세) 초안 생성
- 활용 원칙: **"생성된 코드는 100% 직접 읽고 합리화한 후 채택"** — AI 도구가 만든 코드라도 책임은 엔지니어에게.

**스피커 노트**:
"비용 최적화에 신경 썼다"를 말로만 하지 않고, **다음 PR에서 얼마를 깎았는지 볼 수 있는 계측 인프라**를 먼저 깔았다는 점이 포인트입니다. 그리고 Claude Code는 본 발표를 포함해 거의 모든 단계에서 사용했고, 그 사실을 숨기지 않습니다.

## 슬라이드 15 — 솔직히 못 한 부분 — 그리고 첫 90일에 할 것

**헤드**: 보강해야 할 4가지 — 그래서 합류 후 첫 90일은 이렇게 쓰겠습니다

**미흡 항목과 그 이유**:

| # | 못 한 부분 | 이유 / 현재 상태 |

| ③ | **다중 평가자 협업·권한 모델 미구현** | 단일 사용자 시나리오에 집중. 평가자별 권한, 교차 평가, 합산 합격 결정 워크플로우는 미구현. |
| ④ | **LLM 평가 자동화(Eval) 부재** | Ragas/LangSmith 같은 평가 도구로 모순 탐지·꼬리질문 품질을 정량화하지 못함. 현재는 사람 검수에 의존. |

**합류 후 첫 90일 액션 플랜**:
- **Month 1** — 위 ①을 해소. 고정 이력서 픽스처 5종 × 계약 검증을 pytest 스위트로 구축, CI에 연결.
- **Month 2** — 위 ②를 해소. Anthropic prompt cache를 시스템 프롬프트·가치 정의·OCR 지침에 적용 → 입력 토큰 비용 분기별 30% 절감 목표를 `token_usage` 테이블로 추적.
- **Month 3** — 위 ④를 향한 첫 걸음. 모순 탐지·꼬리질문에 대한 **회귀용 골든 데이터셋 30건** 구축, 자동 평가 파이프라인 초안.

**스피커 노트**:
지금 이 슬라이드는 자랑이 아니라 정직한 점검입니다. **무엇을 못 했는지 알고 있다는 것**과 **그것을 어떤 순서로 풀 것인지가 정해져 있다는 것**이 — 그 사람을 뽑는 이유라고 생각합니다. 미흡 항목과 90일 액션이 1:1로 매핑되어 있습니다.
